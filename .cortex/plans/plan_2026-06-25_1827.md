## Context

- This repo is a demo workspace with sample PRD files (CSV) describing source onboarding, business rules, and column mappings for an AP invoices pipeline.
- There's an existing plan for `SILVER_AP_INVOICES` that follows a pattern: parse PRD → extract mappings/rules → produce a structured implementation plan for a Dynamic Table.
- No `.cortex/skills/` directory exists yet — we'll create it from scratch.
- The skill should accept XLSX files (primary) as input, since that's the most common PRD delivery format from business stakeholders.

## Implementation Steps

### 1. Create directory structure

```
.cortex/skills/prd-to-dynamic-table/
├── SKILL.md              # Main skill file (workflow, stopping points, output spec)
└── scripts/
    └── parse_prd.py      # Python script to extract sheets from XLSX into structured JSON
```

### 2. Write `SKILL.md`

The skill will contain:

- **Frontmatter**: name `prd-to-dynamic-table`, description with trigger phrases
- **When to Use**: PRD/requirements docs that define source-to-target mappings for a Dynamic Table
- **Inputs**: `prd_path` (path to XLSX file), `target_dynamic_table` (fully-qualified table name like `DB.SCHEMA.TABLE`)
- **Workflow** (5 steps):
  1. **Parse** — Run `parse_prd.py` to extract sheets into structured data (source onboarding, column mappings, business rules)
  2. **Catalog lookup** — Use `cortex search table-details` to inspect the target DT's current schema (or confirm it doesn't exist yet)
  3. **Analyze** — Identify new sources, new/changed columns, transformations, and business rules. Classify each as Confirmed, Open, or Assumption.
  4. **Surface open questions** — Present ambiguities and assumptions to the user via `ask_user_question` before proceeding. Never guess.
  5. **Produce output** — Generate the structured output document (see Output section)
- **Stopping Points**: After Step 1 (confirm parsing was successful), after Step 4 (user resolves open questions)
- **Output specification** (always returned):
  - New source systems (table)
  - Column-level mapping changes (additions, type changes, transforms)
  - Business rules affecting Silver (with rule IDs)
  - Assumptions made (explicit list)
  - Open questions requiring human decision
  - Recommended DDL sketch (CREATE OR REPLACE DYNAMIC TABLE with UNION ALL branches)
- **Best practices section**: "Surface assumptions, don't guess" rules — if a mapping is ambiguous, the skill MUST stop and ask rather than infer

### 3. Write `scripts/parse_prd.py`

A Python script (runnable via `uv run`) that:
- Accepts `--input <path.xlsx>` argument
- Uses `openpyxl` to read all sheets
- Heuristically identifies sheets by header patterns (looks for columns like "Source System", "Source Column", "Target Column", "Rule ID", etc.)
- Outputs structured JSON to stdout with three sections: `source_onboarding`, `column_mappings`, `business_rules`
- Handles common XLSX quirks (merged cells, empty rows, inconsistent headers)
- Falls back gracefully with a clear error if a sheet can't be parsed

### 4. Add inline `pyproject.toml` dependency

The script will use a PEP 723 inline metadata block (`# /// script`) so `uv run` can resolve `openpyxl` without a separate project config.
