## Context

- Both bronze tables have 17 columns each with 1:1 semantic mappings (different names, same data types)
- Payment terms need normalization: SAP `NET30`/`NET60` vs Oracle `N30`/`N60`
- Approval status needs normalization: Oracle `VALIDATED` mapped to `APPROVED` (most conservative assumption given both systems treat it as "ready to pay")
- Source-specific metadata columns (`SAP_COMPANY_CODE`, `SAP_DOCUMENT_TYPE`, `ORACLE_ORG_ID`, `ORACLE_SOURCE`) are intentionally excluded from Silver — they don't generalize across systems
- A Dynamic Table with `TARGET_LAG = '1 hour'` on `COCO_WORKSHOP_WH` keeps the Silver layer auto-refreshing

## Implementation Steps

**Single DDL statement** — `CREATE OR REPLACE DYNAMIC TABLE COCO_WORKSHOP.PIPELINE_LAB.SILVER_AP_INVOICES`:

1. **Surrogate key**: `MD5(SOURCE_SYSTEM || '|' || SOURCE_INVOICE_ID)` as `INVOICE_SK`
2. **Source discriminator**: Literal `'SAP'` or `'ORACLE'` as `SOURCE_SYSTEM`
3. **Canonical columns** (17 total): `SOURCE_INVOICE_ID`, `INVOICE_NUMBER`, `VENDOR_ID`, `VENDOR_NAME`, `INVOICE_DATE`, `DUE_DATE`, `INVOICE_AMOUNT`, `CURRENCY_CODE`, `PAYMENT_TERMS_RAW`, `PAYMENT_TERMS` (normalized), `PO_NUMBER`, `LINE_DESCRIPTION`, `GL_ACCOUNT`, `COST_CENTER`, `APPROVAL_STATUS_RAW`, `APPROVAL_STATUS` (normalized), `CREATED_AT`
4. **Normalization via CASE**:
   - `PAYMENT_TERMS`: `N30` → `NET30`, `N60` → `NET60`
   - `APPROVAL_STATUS`: `VALIDATED` → `APPROVED`
5. **Raw columns preserved**: Both `*_RAW` and normalized versions are kept so nothing is hidden

## Verification

- `SELECT COUNT(*) FROM COCO_WORKSHOP.PIPELINE_LAB.SILVER_AP_INVOICES` — expect 30 rows (15 SAP + 15 Oracle)
- `SELECT SOURCE_SYSTEM, COUNT(*) ... GROUP BY SOURCE_SYSTEM` — expect 15/15 split
- `SELECT DISTINCT PAYMENT_TERMS` — expect only `NET30`, `NET60`
- `SELECT DISTINCT APPROVAL_STATUS` — expect `APPROVED`, `PENDING`
- Verify no NULLs in `INVOICE_SK`
